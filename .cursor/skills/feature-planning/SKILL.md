# Feature Plan — 单功能分层实现思路

## 触发时机

**进入任意后端功能（含基础设施）的开发前**，由模型主动触发本 skill。

**开始时须声明**：「我正在使用 feature-plan skill 为「[功能名]」生成分层实现思路。」

---

## 核心目标

本 skill 的目的不是把完整代码和完整实现清单提前写进 `.output/plan.md`，而是：

- 给模型提供**规范的开发路线图**
- 让学生能看懂**这一功能会按什么顺序开发**
- 让用户在动工前确认**功能边界、分层顺序、测试入口**

---

## 强制要求

生成内容时必须遵守以下规则：

- **只写大致实现思路，不写完整代码块**
- **只写分层目标，不写逐行实现清单**
- **只写功能级测试思路，不写冗长的命令大全**
- **语言必须让学生看得懂**，避免只堆技术术语
- **不得出现**「TBD / 后续补充 / 先这样 / 到时再看」等模糊占位

---

## 第一步：读取功能信息

从 `.output/plan.md` 中读取并整理：

- 本功能的**业务目标**
- 本功能的**依赖功能**
- 本功能对应的**前端页面 / 测试入口**
- 已确认的 **Python 指令**
- 与本功能相关的**配置项 / 外部依赖**

若以上信息缺失，先补齐再生成思路。

---

## 第二步：预估涉及文件

先给出本功能预计会新建或修改哪些文件，但只要求到**文件级别**，不展开到代码级别。

示例：

```text
预计新建：
  backend/src/models/xxx.py
  backend/src/repositories/xxx.py
  backend/src/services/xxx.py
  backend/src/api/routes/xxx.py

预计修改：
  backend/src/db/models.py
  backend/src/main.py
```

要求：

- 文件路径尽量具体
- 可以标注“预计”“如需”
- 不要求提前写出完整导入、完整类名或完整代码片段

---

## 第三步：生成分层实现思路

按以下顺序写出**每层要实现什么目的**、**这一层的产出是什么**、**为什么要先做这一层**。

默认顺序：

```text
models（Pydantic） → db/models.py（ORM） → repositories → services → api/routes → 前端联调
```

每一层建议使用以下结构：

```markdown
**层 1 — Pydantic 模型**
- 目标：定义这个功能的请求 / 响应数据结构，先把接口契约说清楚。
- 主要产出：创建 / 更新 / 列表 / 详情等模型（按功能需要决定）。
- 说明：这一层先定好字段，后面的 ORM、Service、路由都围绕这套契约展开。
```

写法要求：

- 用**白话解释层的作用**
- 说清楚**这一层完成后，下一层为什么能继续做**
- 允许写“预计包含哪些能力”，但不要写完整方法清单
- 若某层本功能**不需要**，可写“本功能暂不需要此层的特殊处理”

---

## 第四步：补充联调与测试思路

计划中必须包含以下两部分，但保持**高层、简洁**：

### 1. 前端联调

说明：

- 这个功能完成后，前端哪个页面会接入真实后端
- 哪些 Mock 接口 / Mock 数据需要同步替换或修正
- 用户最终从哪个页面、通过什么操作测试这个功能

### 2. 用户测试思路

只需写清楚：

- 启动入口是什么
- 用户在页面上如何触发
- 成功时大概会看到什么
- 出问题时主要看哪里排查（如日志、接口返回、页面提示）

**不要**在 plan 里塞入大量 curl、查库命令、逐层验证命令；这些属于实际开发完成后的测试引导，再按当时功能状态提供给用户。

---

## 第五步：写入 `.output/plan.md`

将分层实现思路**直接写入 `.output/plan.md`** 对应功能下方，采用下面这种简洁结构：

```markdown
### [序号]：[功能名]（依赖 [依赖项]）

> 分层实现思路由 feature-plan skill 生成，用户确认后动工。

#### 功能目标
- 用一句白话说明这个功能要让用户得到什么结果

#### 预计涉及文件
- 新建：`backend/src/models/xxx.py`
- 新建：`backend/src/services/xxx.py`
- 修改：`backend/src/db/models.py`
- 修改：`backend/src/main.py`（如需）

#### 分层实现思路
- [ ] **层 1：Pydantic 模型**
  - 目标：先定义接口契约，让前后端对字段和数据结构达成一致。
  - 产出：本功能需要的请求 / 响应模型。

- [ ] **层 2：ORM 表结构**
  - 目标：补齐这个功能需要落库的数据结构。
  - 产出：对应表、字段、关系或索引。

- [ ] **层 3：Repository**
  - 目标：把本功能需要的数据读写动作收口，避免业务层直接操作数据库。
  - 产出：本功能需要的基础查询 / 新增 / 更新能力。

- [ ] **层 4：Service**
  - 目标：承接业务规则、状态判断和异常处理。
  - 产出：供路由调用的核心业务方法。

- [ ] **层 5：API 路由**
  - 目标：把前端会用到的接口暴露出来，并接入统一响应。
  - 产出：本功能对应的接口入口。

- [ ] **层 6：前端联调**
  - 目标：将前端对应页面从 Mock 切到真实后端。
  - 产出：修正后的接口调用、页面反馈与联调结果。

#### 用户测试思路
- 测试入口：`[页面名称 / 操作入口]`
- 触发方式：`[用户做什么操作]`
- 成功表现：`[页面/接口/日志的大致预期]`
- 排查重点：`[优先看日志、接口返回或页面提示]`

- [ ] **用户测试通过（本人执行后勾选）**
```

---

## 第六步：自检

写入前检查：

- [ ] 是否写成了**路线图**，而不是完整代码实现
- [ ] 是否说明了**每一层的目标**，而不是堆术语
- [ ] 是否写明了**前端联调与 Mock 替换**
- [ ] 是否让学生能看懂**开发顺序和测试入口**
- [ ] 是否避免了过长命令、代码块和实施细节

---

## 第七步：等待用户确认

写入完成后，对用户说：

**「『[功能名]』的分层实现思路已写入 `.output/plan.md`。**

**摘要：**
- 涉及文件：[列出 3-6 个关键文件]
- 开发顺序：Pydantic → ORM → Repository → Service → Routes → 前端联调
- 测试入口：[页面或接口入口]

**请确认：是否按此思路开始开发？（回复“确认”或提出修改意见）」**

**在用户明确回复确认之前，不得开始写任何代码。**
