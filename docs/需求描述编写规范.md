# 需求描述编写规范

> 适用于本仓库 **功能需求说明**（产品验收源稿）与 **标注文档**（研发落页）的编写与维护。  
> 范例：[v1.4.0/功能描述-交期与订单运营-v1.4.0.md](../v1.4.0/功能描述-交期与订单运营-v1.4.0.md)

---

## 1. 文档分层

| 层级 | 文件 | 读者 | 写什么 | 不写什么 |
|------|------|------|--------|----------|
| L1～L3 | `功能描述-{模块}-v{x}.md` | 产品、业务、前端、测试 | 边界、流程、卡片规格（中文） | 研发标识、接口字段名 |
| L4 | `annotation-docs/*.md` + `js/annotation-spec-data.js` | 开发、标注落页 | 标识映射、三栏摘要 | 重复抄 L3 长文 |

**单一事实来源**：卡片细节以 **功能需求说明 §5** 为准；标注层只摘录 + 映射。

---

## 2. 功能需求说明 · 固定七章

每个模块一份，文件名：`功能描述-{模块名}-v{版本}.md`。

| 章节 | 标题 | 必填内容 |
|------|------|----------|
| **0** | 文档元信息 | 版本、状态、依赖链接、变更记录（表格） |
| **1** | 一句话与边界 | 做什么 / 不做什么 / 完成标准（可勾选验收） |
| **2** | 角色与前置 | 谁用、须先选客户、门禁规则 |
| **3** | 流程总览 | 流程图 + **链路表**（见 §3.1） |
| **4** | 分支与异常 | 无数据、门禁、话术直达、toast |
| **5** | 卡片规格 | 按链路顺序；每张卡用 **§5.0 六块模板** |
| **6** | 数据与状态 | 业务名：订单类型、状态流转、列表筛选规则 |
| **7** | 版本差异 | 仅列与其它版本的 delta；正文写当前版本真相 |

### 3.1 链路表模板

| 步骤 | 用户动作 | 界面（中文名） | 下一跳 | 可跳过 |
|------|----------|----------------|--------|--------|
| 1 | … | … | … | … |

### 5.0 单张卡片 · 六块模板

每张卡固定顺序，禁止混用「字段说明」「交互逻辑」等不同标题：

| 块 | 写什么 |
|----|--------|
| **出现** | 触发方式、前置条件 |
| **展示** | 标题、区块、只读/可编辑、列表列 |
| **操作** | 按钮/点选 → 结果 |
| **数据从哪来** | 取数范围、排序、筛选（业务语言） |
| **校验与提示** | toast、置灰、对话提示 |
| **关联** | 与其它卡关系；共用宿主写 **场景差异表** |

**共用宿主**（同一张选单卡多场景）：在 §5 写一张「订单选单卡」，用子表列复制/变更/进度/交期自选差异，不要复制四份正文。

## 3. 标注文档（L4）· 固定结构

`annotation-docs/{编号}-{模块}.md`：

1. **模块链路表**（从 L3 复制，增加「标注标识」列）
2. **标识映射表**（中文模块名 ↔ 宿主说明）
3. **三块摘要表**（见下表；与 `annotation-spec-data.js` 同源）

`annotation-spec-data.js` 每张卡写在 **`content` 数组**内，按文档体分块；`query` / `interaction` **留空**（由 `content` 统一承载）。验收面板 **不写英文标识与代码名**（界面按钮原文除外）；**不写演示数据来源、mock、正式对接等实现环境描述**。

### 3.1 单张卡片 · 三块模板（L4 固定顺序）

| 块 | 写什么 |
|----|--------|
| **一、展示与口径** | 界面怎么呈现 + 各指标/字段含义与计算规则；**父项 `1.` 写展示，子项 `1.1.` `1.2.` 写对应口径** |
| **二、范围** | 数据范围、是否须选客户、权限与过滤边界（可无此块） |
| **三、交互** | 入口、引导语、操作结果、负向约束、异常提示 |

**写法约定**：

- 块标题用 `【一、展示与口径】` … `【三、交互】`
- 展示与口径同一章：顶层 `1.` `2.` 为 UI/区块；其下 `1.1.` `1.2.` 为该项口径
- 范围、交互块内用 `1.` `2.` 编号
- L4 摘要表三列：**一、展示与口径 | 二、范围 | 三、交互**（不用「内容/查询/交互」）

细则见 [.output/标注编写规范.md](../.output/标注编写规范.md)。

## 4. 写作约定

1. **产品稿用业务名**：写「未审核」，不写内部状态字段名。
2. **先流程后字段**：§3 链路表未完成，不得写 §5 卡片细节。
3. **可验收**：§1 完成标准须能转为测试步骤。
4. **版本只写在 §7**：正文仅描述当前实现版本。
5. **引用不复制**：流程一致时写「§3～§6 同 v1.3.0，差异见 §7」。
6. **变更记元信息**：§0 变更记录表格，避免只查 git。

## 5. 与现有文件关系

| 现有 | 迁移后 |
|------|--------|
| `功能描述-*.md`（旧 §1.5 编号） | 按七章重排；去掉正文中的研发标识 |
| `annotation-docs/*.md` 长文 | 缩为 L4 三部分 |
| `v1.x/PRD.md` | 页面索引 + 链接到各模块需求说明 |
| `功能描述-方案报价下单` | 主链路模块；运营类只链接不重复 |

## 6. 维护流程

1. 改需求 → 先改 **功能需求说明 §3～§5**
2. 从 §5 摘录更新 **annotation-docs** 三栏摘要
3. 同步 **annotation-spec-data.js**（仅改对应标识）
4. §0 变更记录加一行
